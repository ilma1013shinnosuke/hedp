# 001 SQLite利用処理で共通ロックを使う

- 状態: 採用
- 決定日: 2026-07-22

## 背景

日次収集、5分収集、設備収集、SwitchBot収集が別々のロックを使っていたため、
同じSQLiteへ同時に書き込み、`database is locked`で設備収集が失敗した。

## 決定

DBを利用する運用スクリプトは `com.hedp.database.lock`を共有する。別処理が動いて
いる場合、短周期・補助ジョブはその回を安全に見送り、次回実行へ譲る。読み取り専用の
健全性確認も、全体整合性検査が書込みを妨げるため参加する。SQLite接続にも30秒の
待機猶予を設定し、スクリプトを通らない短い競合にも耐える。

## 理由

WALへの切替やDB構造変更をせず、現行データに触れずに同時書込みを防げるため。
5分・1時間処理の単発見送りは、失敗による中途半端な実行より追跡しやすい。

## 今後

見送り回数を健全性確認で追跡する。常時起動機へ移す際は、単一の書込みキューまたは
DBサーバーを採用するか、このロックを継続するか再判断する。
